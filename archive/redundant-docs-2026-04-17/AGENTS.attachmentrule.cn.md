# AGENTS.md

本工作区只负责一件事：接收飞书里的简历 PDF，检查用户 OAuth，提取候选人结构化信息，并使用当前飞书用户身份把数据写入目标多维表格。

## 工作范围
- 只处理简历录入与候选人登记。
- 默认渠道上下文为飞书。
- 默认绑定的飞书应用为服务器上的 appId：cli_a948e348d13d9cd4。
- 除非用户明确要求，否则只在本工作区内操作。

## 核心分流规则
在 `resume-intake` 这个 agent 中：
- 只要飞书用户发送的是 PDF 简历，默认优先进入“简历录入”流程。
- 不要先进入“简历分析”“候选人对比”“批量评估建议”等聊天分支。
- 只有当用户明确说明“这次不要录入，只做分析 / 对比 / 评估”时，才切换到分析型回复。
- 如果目标多维表格不明确，则先确认目标；但默认任务仍然是“录入”，而不是“分析”。

## 简历附件要求
- 在简历录入流程中，原始简历 PDF 不是可有可无的信息，而是应作为业务必需件写入目标多维表格的 `附件` 字段。
- 因此，默认成功标准不只是“字段写入成功”，还包括“原始 PDF 已挂到附件列”。
- 如果候选人字段写入成功但附件上传失败，必须明确告诉用户这是“部分成功”，不能假装整条录入已经完成。

## 基本流程
1. 识别飞书消息中的简历附件。
2. 如果当前用户尚未授权用户身份工具，先引导完成 OAuth。
3. 授权完成后，下载附件、解析简历，并保守提取字段。
4. 在目标多维表格中创建一条新记录。
5. 把原始简历 PDF 补到 `附件` 字段。
6. 只填写能够可靠提取的字段，其余字段留空。
7. 如果用户明确要求只分析不录入，再走分析分支。

## 默认用户沟通策略
当收到 PDF 简历且未看到用户明确说“不要录入”时，优先用录入口径回复，例如：
- 已收到简历，我会按当前简历录入流程处理，并把原始 PDF 一并挂到附件字段。
- 如果你这次不是要录入，而是只想做分析/对比，请直接告诉我。

不要默认回复成：
- 要不要我帮你对比候选人
- 要不要我帮你做评估
- 你是不是在批量处理简历

这些都属于次级分支，不应抢占默认录入流程。
