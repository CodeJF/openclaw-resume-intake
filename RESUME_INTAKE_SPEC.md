# 简历录入规范 v1

## 产品目标补充
当飞书用户上传 PDF 简历并进入录入流程时，成功标准包括两部分：
1. 候选人字段写入目标多维表格；
2. 原始简历 PDF 同步写入目标记录的 `附件` 字段。

## 附件字段要求
- `附件` 字段属于该业务链的必需部分，不是可选增强项。
- 默认业务完成标准应包括附件上传成功。
- 如果字段已写入但附件未成功写入，必须向用户明确说明这是“部分成功”。

## 稳定写入顺序补充
推荐顺序调整为：
1. 接收飞书里的简历 PDF；
2. 下载文件；
3. 提取文本并生成字段；
4. 先创建记录；
5. 再将原始简历 PDF 写入 `附件` 字段；
6. 向用户反馈最终结果。

## 用户可见反馈要求
- 字段写入和附件写入都成功：回复完整成功。
- 字段写入成功但附件失败：回复部分成功，并点明附件失败。
- 字段写入失败：回复失败原因，不要沉默。


## 附件补传标准链路
当前附件链路的标准顺序为：upload -> file_token -> guarded attachment update -> record.update。
可通过 `scripts/resume_intake_attachment_pipeline.py` 查看标准动作顺序。


## create 后不得直接结束
在当前正式流程中，create 成功后必须继续执行附件补传；不得在 create 返回 record_id 后直接结束并向用户回复“已录入”。

判定规则：
- create 成功 + 附件 update 成功 = 完整成功
- create 成功 + 附件 update 失败 = 部分成功
- create 成功 + 未执行附件补传 = 流程不完整，视为 bug


## 正式执行链要求
真实执行链必须遵守：create -> upload pdf -> get file_token -> update 附件 -> reply。
任何在 create 成功后直接结束并回复用户的行为，都视为流程错误。
